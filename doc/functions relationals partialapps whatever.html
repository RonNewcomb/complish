This doc does NOT mean using the relative clause method of embedding an expression within another. Rather it means replacing a parameter with an unbound type, like it was declaring rather than invoking: add 5 to a number.

This doc DOES assume that we're in an invocation context, i.e., inside the function body.

When invoking a relation during a body:

  • A relation is either a function...
    A person sells a car by...
    ...or a list of structs.
    A person loves a person.
  • A relation can either stand alone as a statement...
    Ron sells the Buick.
    ... or be embedded within a larger statement as an expression...
    Ron learns Ron selling the Buick at college.
  • A relation can either be provided all its arguments...
    Ron sells the Buick.
    ...some of them...
    A person sells the Buick.
    ...or none of them.
    A person sells a car.

A partially applied relation, as a statement, can be joined to another via a subordinating conjunction to form a constraint. But any bound parameters must refer to an unbound parameter elsewhere in the sentence.

How to deal with imperatives outside of a sentence containing "to verb"?

This breaks down into:

  1. how to know a function definition is beginning that will span multiple sentences.
  2. how to determine where the multi-sentence imperative function definition ends.

For #1, a continuation of an existing imperative: "To drive a car, get the keys. Open the door and enter the car. Shut the door." If a sentence is imperative, by itself, like "Verb the noun.", then it obviously must connect to something; it's obviously a step in a series of steps. So connect it to whatever sentence is in front of it? What if the preceding sentence doesn't make any sense?

For #2, the fact that any naked imperative must connect to something previous (aside from parenthetical asides), and anything leftover must end it, by process of elimination. Well, if a statement only makes sense inside a function body, assume it belongs to the most recent one no matter how many sentences ago one was. (Well, check and see if it actually uses any of the outputs of them; if not then error.) I don't see any real problem with allowing struct definitions "mid-body". Heck, maybe they are intended to be local to the function.

Variables and Their History

How do variables with a past-participial verb in their name, like "the gotten car" or "the drawn wagon", work?

  1. The variable is a global and the verb was the function in which it appeared?
  2. "apply this single-input single-output function to this noun right now"?
  3. "this was the function that was previously applied to this noun"? But what about functions which need be applied multiple times?

Declarations use adjectives as type: the textual title, the numeric length, etc. Can a verb, translated into an adjective via the past participle, be used this way? Say there's a function called sanitize which removes injection attacks or whatever. So the adjective would be sanitized, and a database-calling function might accept "a sanitized text" rather than just "a text".

Well are the basic types actually a verb? No, "numeric" isn't a past tense verb. There *is* a verb "to number", but the type would be "a numbered list", not "a numeric list".

The above presumed option #3, the function was previously applied. But for sanitized text it seems to work at that moment as well. Should the code even make a distinction? Maybe sanitize will be applied to the text if it has not already been? That would make sense, and it dovetails with the way lazy evaluation tends to create structures like 2+(3*(4/5))) instead of evaluating them strictly; here the history is checked to ensure text is sanitized.

Does the notion of variable history extend to encompass immutable-once-assigned variables and imperative variables? (Not really?)
